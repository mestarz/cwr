# 问题/需求视图

## 核心规则：向用户展示某一个需求、问题时，需要展示完整视图；同时控制在 300 字内

- 问题所属框架、场景
- 问题的原因、导致的问题
- 不同解决方案的优劣

# 代码文档编写准则

- 代码注释、文档中必须以中文为主
- 不同逻辑的代码、不同类别的数据结构定义需要拆分，模块要解耦
- 不要过度抽象，对于不必要的场景不要增加额外中间层
- 不要擅自增加兜底错误处理逻辑
- 不要实现显而易见的测试代码，测试代码需要和源码隔离

# 极简开发模式

- 当前默认身份是极简主义开发者，优先实现核心功能逻辑并保持最高可读性。
- 未经开发人员明确要求，禁止添加 try-catch、默认值、冗余 if-else、长参数检查等防御性代码。
- 输入或数据不符合预期时允许程序早期抛错，错误应向上暴露，不捕获并忽略。
- API 或函数返回为空时，不伪造空对象、空字符串或假成功结果。
- 主动清理废弃变量、函数、导入、类和旧逻辑注释；函数职责过多时拆解并删除臃肿实现。
- 代码应直达业务主干流程，语义化命名，合理模块化，只为复杂算法写必要注释。
- 输出代码块前先声明：`[Lean Mode Active]: 仅实现核心逻辑，未添加额外防御性代码。`

# 项目独有规则记录

- 当前项目独有、长期有效的约束，应记录到仓库根目录 `rules/` 下。
- 每个规则文件使用 Markdown，文件开头前 3 行必须详细说明规则涉及方向。
- 规则正文保持精简，只写可执行约束，不写大段背景。
- `AGENTS.md` 可保留总入口和通用执行要求，细分项目规则沉淀到 `rules/`。

# Emma Web 编辑器开发规范

- 当前项目目标是 Web 端 Emma 编辑器，前端使用 TypeScript、React、Vite。
- 一期优先实现文本输入与 Agent 交互，不提前实现语音链路。
- 前端页面核心区域包括：中间代码展示区、左侧 AI 解释区、Markdown/图展示区、相关文件列表、文本输入区。
- 后端 Agent 只能使用 Pi Agent：`https://github.com/earendil-works/pi`。
- Pi Agent 集成必须优先使用 TypeScript Coding Agent 包 `@earendil-works/pi-coding-agent`，不得改用其他 Agent 框架。
- Emma 后端不得自行选择或覆盖 AI 模型；默认模型、A 键认证、ModelRegistry 和 AuthStorage 由 Pi Coding Agent 自己处理。
- 不调用本地 Pi Agent CLI 或本地 Agent 进程作为一期集成方式，除非开发者后续明确要求；当前目标是直接在 TypeScript 后端中集成 Pi Agent 能力。
- Agent 交互逻辑应通过本地 TypeScript 后端或 Agent Bridge 承接，浏览器端不得直接暴露密钥或直接运行需要服务端权限的 Agent 能力。
- AI 返回内容优先结构化表达，至少区分解释文本、Markdown 内容、相关文件、代码定位信息。
- 代码展示优先采用成熟编辑器组件，不手写复杂代码高亮、滚动同步、行号等基础能力。
- AI 解释、Markdown 图等展示型 UI 不是常驻面板，只能由 Agent 调用对应 skill 后显示为右侧悬浮框。
- 服务端展示型 skill 统一放在 `server/skills/` 下，前端只消费 skill 产出的结构化 `floatingWidgets`，不得直接耦合具体 Agent 实现。

# 前端编写规范

- 禁止使用 emoji / 表情符号，避免界面表达依赖装饰性符号。
- 禁止使用紫色或紫蓝色作为主视觉主题。
- 默认采用浅色主题，除非需求明确指定其他主题。
- 图标统一使用简洁、克制、一致的风格，避免彩色杂乱图标。
- 工具类按钮优先使用图标加 tooltip，不在紧凑工具栏中堆叠大段文字。
- 页面应直接呈现可用编辑器界面，不做营销式落地页。

## 任务进度记录

每完成一个小任务，必须在 `/tmp/cwr/` 目录中记录进度。

### 目录结构

```text
/tmp/cwr/
├── progress.md        # 主进度日志（追加写入）
└── tasks/
    └── <task-id>.md   # 单个任务的详细记录（可选）
```

### 记录格式（追加到 progress.md）

```markdown
## [YYYY-MM-DD HH:MM] <任务简述>

- **状态**: done / in-progress / failed
- **修改文件**: src/foo.ts, src/bar.tsx
- **变更摘要**: 简要描述做了什么
- **验证结果**: lint 通过 / tsc 通过 / 测试通过
```

### 初始化记录目录

若 `/tmp/cwr/` 不存在，在首次记录前执行：

```bash
mkdir -p /tmp/cwr/tasks
```

## 子 Agent 完成后的 Code Review

子 agent 每次修改后，主 agent 必须执行 git diff 检查：

```bash
# 查看所有未提交的变更
git diff

# 查看暂存区变更（如有）
git diff --cached

# 查看变更文件列表
git diff --name-only
```

检查要点：

1. 变更是否与任务目标一致（无无关改动）
2. 是否引入潜在 bug 或类型错误
3. 是否有敏感信息（密钥、密码等）被提交

确认无误后再进行下一个任务。

## Commit 规范

### Commit Message 格式

- 使用中文描述提交内容
- 格式：`<类型>: <简述>`

| 类型       | 说明               |
| ---------- | ------------------ |
| `feat`     | 新功能             |
| `fix`      | 缺陷修复           |
| `refactor` | 重构（不影响功能） |
| `style`    | 代码格式/样式调整  |
| `chore`    | 构建/配置/依赖变更 |
| `docs`     | 文档更新           |

### Commit 前必备文档

每次 commit 前必须在 `docs/issues/` 下新增或更新一份说明文档。
该文档应记录本次修改的背景、目标、实现内容、修改文件清单与验证结果。
该文档的状态字段应在提交前设置为 `done`，并与代码变更一并纳入同一次提交。

- 文件命名遵循约定：`NNN-{type}-{slug}-{YYMMDD}-{status}.md`，其中 `NNN` 为下一个编号，`type` 用 `feature`/`fix`/`refactor`/`docs`/`chore` 等，`status` 用 `wip`/`done`
- 文档应至少包含：标题、元信息表格（编号/类型/状态/时间）、背景、目标、实现内容、修改文件清单、验证结果
- 提交前使用 `git diff` 和 `git status` 检查变更内容，确认 issue 文档状态已改为 `done`
- commit message 中应引用该 issue 编号，例如：`feat: 新增 Emma 编辑器基础界面 (refs #001)`

## 受保护的核心接口

Pi Agent Bridge、server skills、前端状态模型、AI 返回结构是当前项目核心接口。

- 未经开发者明确允许，禁止随意重命名或删除这些接口中的字段。
- 若实现某个功能必须调整 schema，应先停下来询问开发者并取得同意。
- 修复纯文本注释、补充字段说明等不改变结构本身的改动可以直接做，但需要在文档中明确说明。
